Product Context and Rationale

The goal is a secure, self-hosted way for one operator to monitor and eventually control Herdr-managed AI agents from desktop and mobile browsers. The deployment targets at most ten hosts and tens of active agents.

Why the old design was replaced

The repository previously centered on a Python relay, SSH or inbound host polling, Telegram integration, native clients, shared URL tokens, and host lifecycle controls. Those paths increased credential exposure, platform maintenance, replay ambiguity, and routing complexity. They also did not provide the exact state preconditions required for safe remote writes.

Enterprise v1 replaces that design with a central Go control plane, outbound-only Go connectors, a same-origin TypeScript PWA, strict JSON contracts, OIDC authentication, mTLS host identity, and metadata-only durable storage.

Confirmed product boundaries

In Scope

  • One MFA-authenticated operator.
  • Responsive installable PWA.
  • Agent status, bounded output, transient prompts, text, safe keys, and interrupt.
  • Outbound-only connectors with mTLS.
  • Durable metadata-only audit and no replay.
  • Web Push as a wake-up hint, never state authority.

Intentionally Out of Scope

  • Telegram, native macOS, native iOS, and current macOS connector work.
  • SSH polling, UDP transport, inbound host listeners, and shared URL tokens.
  • Agent launch or termination, arbitrary shell execution, host power, and session lifecycle.
  • Multi-user RBAC, multitenancy, managed SaaS, and high availability.
  • Terminal transcript or prompt persistence.

POC Success Definition

  • The operator signs in through Authelia MFA.
  • Linux hosts appear without opening inbound ports.
  • Desktop and Android PWA show current agents and bounded output.
  • Checked writes reject stale state and are never replayed.
  • Operations can back up, restore, rotate, revoke, monitor, and roll back.

State and action model

  • The global agent identity is (host_id, instance_id, terminal_id). Mutable pane, tab, and workspace IDs are never global routing keys.
  • A connector reconciliation creates a connector_epoch; a browser projection creates a separate state_epoch.
  • Every write binds to connector epoch, agent generation, nonzero Herdr input revision, agent label, allowed status, and, for prompts, content hash and prompt fingerprint.
  • The control plane stores the action ID before dispatch. Duplicate IDs never dispatch, including after restart or retention cleanup.
  • The connector sends action.received only after final validation and before execution.
  • A write with uncertain side effects becomes unknown/OUTCOME_UNKNOWN. It is never replayed automatically.
  • Reconnects create fresh epochs and full snapshots. Neither peer repairs gaps by replaying old state or writes.

Security and Data Invariants

These are implementation constraints, not suggestions. A change that weakens one requires an explicit protocol and threat-model review.

Browser Boundary

  • Browser listener stays on loopback.
  • Proxy strips and replaces all four identity headers.
  • Opaque cookie is secure, HTTP-only, and same-site.
  • WebSocket upgrades require exact Origin.
  • State-changing HTTP requires Origin, opaque session, and CSRF.
  • Logout revokes every socket for the application session.

Connector Boundary

  • Connector exposes no listener.
  • Every connection uses WSS and verified mTLS.
  • Host identity comes from the certificate mapping.
  • Only one live connector lease owns a host.
  • Private keys are generated and retained on the connector host.
  • Certificate rotation and revocation are application-enforced.

Content Boundary

  • Prompt excerpts, output, text, and keys are transient.
  • Audit rows contain identifiers, operation type, times, outcome, and stable code only.
  • Push payloads carry a random event ID and coarse event kind only.
  • Logs, metrics, backups, URLs, and errors must not contain terminal content or credentials.

Write Boundary

  • Revision 0 is read-only.
  • checked_input.v1 requires exact inspected Herdr schema.
  • Original Herdr 0.7.3 is always read-only.
  • Stale epoch, generation, identity, status, revision, or prompt data rejects before enqueue.
  • Unknown writes are never replayed.

Deployment and POC Incidents

These failures are part of the implementation record. Preserve them in tests and runbooks so later agents do not repeat the diagnosis.

Systemd rejected an encoded logout URL

Symptom: service deployment failed with Failed to resolve unit specifiers ... Invalid slot because systemd interpreted %3A in the encoded URL.

Resolution: changed the Nix module/service argument handling so the configured URL reaches the process without systemd specifier expansion. Commit 215ec10.

Connector started before Home Manager SOPS materialized the key

Symptom: first start failed with open /home/sab/.config/sops-nix/secrets/herdr_connector_client_key: no such file or directory.

Resolution: systemd restarted the connector after SOPS materialization. The user service is now stable. Future deployment tests should explicitly order and verify secret availability.

Host connected but no instances or agents appeared

Symptom: authenticated live snapshot showed hosts: 1, instances: 0, agents: 0 while Herdr itself reported two agents.

Root cause: connector reconciliation subscribed to pane.agent_exited, which Herdr rejected. The supported event is pane.exited. Reconciliation stopped before the first instance snapshot while the mTLS host lease stayed connected.

Resolution: event corrected, subscription regression added, package repinned, and mz reactivated. Live probe then showed one instance and two agents. Commits 8f162e5 and 987926f.

PWA stayed on connecting and waiting for current state

Symptom: the authenticated PWA displayed Connecting to the control plane... and Waiting for current state... even though a direct WebSocket probe received a snapshot.

First cause: browser instances reused the connector snapshot Go type and leaked connector-only sequence: 0. The strict PWA schema rejected that additional property and closed the socket.

Second cause: after the first correction, recovered hosts and zero-agent instances serialized empty collections as null. The browser schema requires arrays, so a fresh post-restart snapshot could still be rejected before the connector reacquired its lease.

Resolution: added a dedicated browser instance type, asserted the exact serialized browser field set, normalized every browser collection to [], and validated the live frame with the production PWA validator both before and after mz reconnected. Commits d0c8727, 6664e6d, 4e2f14a, and c15aeb6.

Documented deploy app did not exist

Symptom: nix run .#deploy.zanoza failed because dotnix exports no such app.

Working command: nix run github:serokell/deploy-rs -- .#zanoza.

Agent Operating Rules

Non-negotiable repository rules

  • Read AGENTS.md, README, relevant protocol docs, and this roadmap before editing.
  • Keep changes minimal and focused. Do not restore legacy relay, Telegram, native clients, SSH polling, UDP, power control, or session lifecycle code.
  • Use Conventional Commits without AI attribution.
  • Push only to sbulav/herdr-remote, sbulav/herdr, and sbulav/dotnix. Do not open upstream PRs.
  • Do not amend, force-push, reset, or modify unrelated work.
  • Never touch or delete the unrelated /home/sab/dotnix/result.
  • Never decrypt or print SOPS values unless the task absolutely requires runtime use. Document secret names and paths, not values.
  • Coordinate before restarting Herdr, the control plane, Authelia, Traefik, or any service that may disrupt current work.

Protocol change rule

A wire-contract change is incomplete until all of these are updated together:

  • Normative JSON Schema under protocol/.
  • Markdown contract under docs/.
  • Go validation and direction handling.
  • Generated TypeScript validator and discriminated types.
  • Positive and negative fixtures.
  • Python, Go, Vitest, and Playwright cases as applicable.
  • Compatibility and rollout notes.

Definition of done for any work package

  1. The task's acceptance criteria are met with evidence.
  2. Relevant tests, lint, type checks, package builds, and git diff --check pass.
  3. Security invariants remain explicit in code and tests.
  4. Docs and runbooks match deployed behavior.
  5. No secret, prompt, output, input, key, or credential body entered Git, logs, metrics, fixtures, or screenshots.
  6. Only intended files are staged and committed.
  7. The fork branch is pushed, deployment is verified if changed, and rollback is known.
  8. The tracker receives an append-only handoff with commits, commands, results, risks, and next dependency.
